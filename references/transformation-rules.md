# 转换规则

## 1. 解析请求

推断请求的行动、用户可见结果、领域、产物类型、已提供的上下文、证据、文件、工具、约束、成功标准、授权级别、缺失信息和冲突要求。

不要要求用户选择任务类别。不要推断超出请求范围的授权。

## 2. 保留不变量

保留明确事实、数值、文件名、标识符、指标、请求的结构、体裁、语气、输出语言、安全限制、证据要求和权限边界。

不要添加用户未提供且来源指南未确立的事实、来源、产品能力、工具、文件、日期、指标、结果或 API 行为。

## 3. 作出最小假设

必要信息缺失时，作出能够生成有用提示词的最小假设。仅当下游模型需要时，才将执行相关假设放入中文提示词的“假设”小节。用中文分别列出假设和缺失事实。

生成提示词前不要提出澄清问题。

## 4. 自适应构建

只使用会实质改变下游行为的段落：

- “角色”：责任或领域上下文有帮助时使用；
- “目标”：说明用户可见结果；
- “上下文”或“证据”：说明已提供的材料；
- “成功标准”：说明完成门槛；
- “约束”：说明安全、业务、证据、权限和副作用限制；
- “工具”：说明前置检索、路由、重试和审批行为；
- “输出”：说明产物形态、语言、长度、结构和语气；
- “停止规则”：说明完成、回退、弃答或所需的最小澄清行为。

不要在简单请求中强行加入所有段落。优先使用以结果为先的措辞和决策规则。只有真正的不变量才使用绝对化表述。

## 5. 根据请求的行动定义自主权

- 回答、解释、审查、诊断或规划：检查并报告；除非用户请求，否则不要实施改动。
- 修改、构建、修复或实现：允许范围内的本地改动和非破坏性验证。
- 对破坏性操作、购买、请求未明确授权的外部写入或实质扩大范围，要求确认。

## 6. 压缩并审计

删除重复规则、冗余示例、不相关工具、泛化角色文本、宽泛风格标签，以及不会改变行为的过程叙述。

检查矛盾、无依据的声明、缺失的证据规则、意外扩大范围、未授权副作用、缺失的验证和缺失的停止条件。

绝不暴露隐藏推理或要求思维链。
